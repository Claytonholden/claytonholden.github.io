2. Executive Summary

This report summarizes my work in CYBR-2102 – Network Defense Fundamentals and documents a complete defense lifecycle: designing a segmented network, monitoring for hostile activity, and responding to an incident using NIST SP 800-61 and CIS Controls. Over the course, I designed a small enterprise-style network in Packet Tracer, captured and analyzed traffic in Wireshark, and wrote an incident response plan for a brute-force RDP attack with attempted data exfiltration.

The final environment demonstrates layered controls at the perimeter, network, and host levels, plus monitoring and change documentation that match how entry-level SOC analysts and network defenders are expected to work in a real organization.

3. System Architecture & Defense Layers

3.1 Logical Network Design

The core design is a three-zone defense-in-depth architecture built in Packet Tracer:

  • External: Simulated Internet / untrusted network.
  • DMZ: 192.168.10.0/24 – public-facing web server and DNS.
  • Internal LAN: 192.168.20.0/24 – user workstations and file server.

A perimeter device (Cisco firewall/router) provides segmentation between the external network, DMZ, and internal LAN. Each interface is assigned to its own subnet and security zone.

3.2 Layered Controls

The design implemented multiple layers of security:

  • Perimeter: Ingress ACLs permit only required traffic (HTTP/HTTPS to the DMZ web server) and drop all other unsolicited traffic.
  • Network: ACLs block direct DMZ–Internal communication, enforce least privilege between VLANs, and restrict management access to specific admin IPs.
  • Host: Local firewalls are enabled on key systems (Admin PC, FileServer) and hardened login banners are configured on Cisco IOS devices.
  • Application/Data: Access-control matrix defines which roles (Admin, IT Support, User) can access the web server, file shares, and management interfaces.

3.3 What Milestone 1 Demonstrates

Milestone 1 shows that I can move beyond a single firewall rule and design a layered architecture where each segment and device has a specific security role and associated controls. The Packet Tracer project includes connectivity tests and ACL verification to prove that the intended policy is actually enforced.

4. Monitoring & Threat Detection

4.1 Tools and Data Collected

For Milestone 2, I used Wireshark to capture several minutes of live traffic in the lab environment. The capture included ICMP, TCP, DNS, ARP, and TLS. To simulate adversary behavior, I generated:

  • An ICMP ping sweep across a large range of IP addresses.
  • A TCP SYN scan against ports 1–20 on the gateway.
  • High-rate ICMP with larger payloads to mimic DoS-style traffic.

4.2 Analysis Approach

I relied on Wireshark display filters to isolate suspicious behavior, for example:

  • icmp to aggregate ping sweeps and ICMP floods.
  • tcp.flags.syn == 1 and tcp.flags.ack == 0 to identify SYN scans.

Each pattern was classified as a true positive (intentionally generated), and I documented how it would look within a SOC workflow: initial alert, verification, and correlation with other data such as source IP and port ranges.

4.3 Mapping to CIS / NIST Controls

  • Rate limiting and filtering ICMP aligns with CIS Control 4 (Secure Network Configurations) and NIST SI-4 (System Monitoring).
  • Detecting port scans and DoS attempts maps to tuning IDS/IPS and log sources as described in CIS Controls 8 and 13.
  • Keeping firewall and router firmware patched ties back to vulnerability management requirements (CIS 7, NIST RA/SA families).

4.4 What Milestone 2 Demonstrates

Milestone 2 shows that I can move from raw packets to detection and recommendations. I can interpret packet captures, identify scanning and flooding behaviors, and suggest concrete configuration changes that would reduce risk in a real environment.

5. Incident Response Summary

5.1 Scenario

Milestone 3 focused on an “Unauthorized RDP brute-force and data exfiltration attempt” against an internal Windows system. The attack was detected via Windows Security log Event ID 4625 (failed logon attempts) and unusual outbound traffic.

5.2 NIST SP 800-61 Lifecycle

Preparation

  • Logging enabled on Windows endpoints and network devices.
  • Incident response roles and communication channels defined.
  • Baseline documentation for normal login and traffic patterns.

Detection and Analysis

  • SOC tooling and log review highlighted repeated 4625 failures from a single source IP.
  • Analysts correlated the failed RDP attempts with outbound connections, indicating possible credential guessing plus attempted data exfiltration.

Containment

  • Disabled the compromised or targeted account.
  • Blocked the attacker’s source IP at the firewall/edge device.
  • Stopped or restricted outbound transfer attempts during investigation.

Eradication

  • Ran anti-malware and integrity checks on the affected host.
  • Removed any suspicious artifacts or tools discovered.

Recovery

  • Restored any impacted data from known-good backups.
  • Returned services to normal operation while monitoring closely for new attempts.

Post-Incident Activity

  • Root cause summarized as exposed RDP with weak authentication and lockout controls.
  • Recommended enforcing MFA, limiting RDP to VPN or trusted networks, and tightening account lockout policies.
  • Mapped recommendations back to CIS Controls (e.g., CIS 4, 6, 16) and NIST SP 800-61 guidance on lessons learned.

5.3 What Milestone 3 Demonstrates

Milestone 3 ties together network monitoring, host security, and backup into a single incident response playbook. It shows that I can describe not only what happened, but how to contain, eradicate, and prevent similar incidents in the future.
